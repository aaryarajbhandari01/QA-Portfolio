# Bug Reports

## Overview
This folder contains documented bug reports discovered during testing. Each bug report is written as a separate file and includes details to help developers understand, reproduce, and fix the issue, and testers to re-test and verify fixes smoothly. 

## Structure
- Each bug is stored in an individual `.md` file.  
- File names follow the format: `DEF-<ID>-<Short Title>.md`.  
- Bug Attachment names the format: `DEF-<ID>-<Short Title>.md`.

## Bug Report Field Descriptions

| Field | Description |
|-------|-------------|
| **Defect ID** | Unique identifier for the issue, e.g., DEF-101 |
| **Title** | Brief summary of the bug |
| **Status** | Current progress: Open, Triaged, In Progress, Fixed, Closed |
| **Severity** | Impact on product functionality: Minor, Major, Critical |
| **Priority** | Urgency for fixing: Low, Medium, High |
| **Platform** | Platform where the bug was observed, e.g., Web, Mobile, iOS, Android |
| **Version** | Application or system version, e.g., v1.4.5, Beta 1.4.5 |
| **Environment** | Environment where the bug occurred, e.g., Dev, Test, Production |
| **Module** | The specific feature or module affected by the bug |
| **User Role** | Role of the user when the bug was encountered, e.g., Learner, New User |
| **Label** | Tags for categorisation, e.g., Bug, Signup, Email Verification, Backend |
| **Related Test Case** | Links to relevant manual or automated tests |
| **Reproduction Steps** | Step-by-step instructions to reproduce the issue |
| **Expected Result** | What the system should have done |
| **Actual Result** | What the system did instead |
| **Attachments** | Screenshots, logs, or design references supporting the defect |
| **Reported On** | Date the bug was documented |
| **Reported By** | Name or ID of the QA analyst who filed the bug |
| **Assignee** | Person responsible for fixing the issue |
